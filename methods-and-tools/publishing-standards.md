
# 多平台运营与技术文章分发指南 (Platform Distribution Guide)

> **说明**：本指南用于指导如何将项目产出转化为全平台影响力，对齐任务书中的内容分发指标。

---

## 1. 平台分类与内容调性 (Platform Matrix)

### A. 平台分类

我们将常用平台按**内容形式与核心受众**重新划分为四大类：

* **技术社区 (Developer Communities)**：V2EX、掘金、思否、开源中国、CSDN、Medium、Reddit、GitHub Discussions。
* **社交/生活/图文 (Social & Visual)**：公众号、小红书、知乎、X (Twitter)。
* **视频/直播 (Video & Live)**：B站、视频号、YouTube。
* **社群/通讯 (Instant Messaging)**：微信群、飞书群、Telegram、Discord。
* **Web 官网 (Web Page)**：Homepage、Docs。

### B. 内容调性 (Platform Matrix)

| 平台类型 | 核心平台 | 推荐内容类型 | 运营目标 |
| :--- | :--- | :--- | :--- |
| **技术社区** | 掘金、思否、c-s-d-n | 深度技术教程、架构设计、pr 记录 | 获取开发者关注、增加 pr |
| **知识社交** | 知乎、公众号 | 行业趋势分析、开源商业化思考、项目周报 | 建立行业影响力、吸引潜在伙伴 |
| **视频平台** | b 站 (bilibili)、视频号 | demo 演示、技术直播剪辑、营员 v-log | 直观展示产品力 |
| **代码社交** | github (discussions) | 路线图 (roadmap)、决策说明、技术文档 | 沉淀核心社区资产 |

## 2. 核心平台运营策略 (Strict Guidance)

### A. 技术社区 (V2EX, 掘金, CSDN 等)

* **定位**：极客与开发者最集中的垂直社区。
* **策略**：严禁无营养的软文。必须以“分享、求助、创造”、“解决具体技术卡点”或“分享开源决策日志”为切入点，直接分享代码逻辑或重大技术决策，回复需专业及时。
* **注意**：V2EX 需注重回帖质量，掘金需注重 SEO 关键词（如 AI-Native, Open Source）。
- **定义**：解决具体技术问题的干货。
- **分发**：首发掘金 -> 转发知乎 -> 代码片段发 github gist。

### B. 视觉与社交 (小红书, 知乎, X 等)

* **定位**：高颜值的信息卡片。
* **策略1**：将枯燥的技术日志提炼为 3-5 个核心 Point，配合精美配图，通过“搞定 xxx 只需要这一招”等钩子进行传播。
* **策略2**：小红书侧重“避坑/保姆级教程”视觉卡片；X (Twitter) 侧重“一句话 Hook + 链接”；知乎侧重“行业深度拆解”。

### C. 动态/视频平台 (B站, 视频号, YouTube)

- **定义**：产品迭代演示、重大决策宣布。
- **分发**：视频发 b 站 -> 动图/短文发全平台动态。
* **策略**：
* **视频号**：侧重 1 分钟内的项目 Milestone 剪辑，适合社交转发。
* **B站**：侧重 5-10 分钟的深度 Demo 演示或技术架构讲解，善用弹幕互动与私域转发。
* **YouTube**：侧重全球流量，5-10 分钟的深度 Demo 演示或技术架构讲解，必须配备英文标题与 CC 字幕，注重长尾 SEO 效果。

### D. 社群管理 (微信, 飞书, Discord, TG)

* **策略**：
* **微信/飞书**：核心在于“温控”，强即时交互，用于发布即时公告和引导用户提交 Bug/Feature、解答营员疑问。
* **Discord/TG**：侧重建立“社区共建者文化”，利用频道隔离不同的技术模块，通过 Role 机制激励核心贡献者。

### E. 官方 Web 资产 (GitHub, Docs, Homepage)

* **定位**：最终的信任背书。
* **策略**：所有平台分发的内容，最终都要有“沉淀地址”回流到 GitHub 或 Docs，是所有外部流量的终点。
* **注意**: Docs 必须保持与代码同步，Homepage 必须展示最新商业化验证成果（如付费意向书）。

### F. 商业/调研类
- **定义**：用户访谈心得、付费意向书背后的思考。
- **分发**：首发公众号/知乎 -> 简版发即刻/朋友圈。

## 3. 内容流转全自动化流程 (The Pipeline)

1. **输入层**：从 `logs/` 提取决策关键点，从 `resources/` 提取 Demo 录屏或截图，`platform-matrix/` 的 Markdown 稿件作为 Source of Truth。
2. **加工层**：
    - **长文加工**：由技术决策记录同步转化为 3000 字左右的技术深度博文、短视频脚本、社群推文。
    - **视频加工**：提取 Demo 核心操作，配以语音讲解，生成 3 分钟内的短视频。
3. **分发层**：利用各平台 API 或分发工具，按“技术社群 -> 社交媒体 -> 官方 Web”的顺序分发。
4. **跟踪机制**:
    - 分发链接携带 UTM 参数，以便在自动化流程中识别流量来源。
    - 格式示例：`https://project-site.com/?utm_source=juejin&utm_medium=article&utm_campaign=launch`
5. **反馈层**：抓取各平台阅读/点赞数据，通过 UTM 标签追踪回流至 GitHub 的转化率，先登记至 `distribution-log.md`，再汇总至 `report/` 周报中。

## 4. 稿件自检清单 (Self-Checklist)

* [ ] 链接是否带了 UTM？
* [ ] 是否针对平台调性调整了第一段话？
* [ ] 敏感词检查（是否带了明显的引流暗语）？
* [ ] 标题是否针对不同平台做了微调？
* [ ] 是否已在文末添加项目 github 链接？
* [ ] 是否已在本地 `community-and-outreach/` 目录下存盘草稿？
* [ ] 是否已在 `report/`（周报）中同步数据？
* [ ] 是否已在 `daily-planner.md` 打钩并关闭 Issue？
* [ ] 核心素材是否已上传至 `resources/` 并能在外部访问？

---

## 5. 通用红线 (General Redlines)

* **禁止二维码**：除公众号文章外，严禁在任何第三方技术平台（V2EX, 知乎, 掘金）正文插入个人二维码。
* **UTM 强制化**：所有对外链接必须使用 `utm_gen.py` 处理，格式：`?utm_source={platform}&utm_campaign={project_name}`。
* **代码块优先**：能用代码说清楚的，不要用截图。

## 6. 平台特定准则 (Platform-Specific Rules)

### 🟢 V2EX (硬核技术地盘)

* **调性**：极度排外广告，但尊重底层技术、极致性能和“独立开发者”精神。
* **死命令**：
* **标题**：必须平实，禁止“震惊体”。
* **正文**：开头必须直奔技术痛点（如：Makepad 如何解决 Rust UI 的内存抖动）。
* **引流**：只能在末尾带一行：`Project Link: [GitHub URL]`。
* **避坑**：发帖后 1 小时内必须回复前 3 个评论，否则会被视为“发完就走”的机器人。



### 🔵 知乎 (长文与专业沉淀)

* **调性**：喜欢深度对比、行业分析、系统性报告。
* **死命令**：
* **结构**：必须有【目录】。
* **配图**：至少包含一张高对比度的系统架构图（用 Excalidraw 绘制）。
* **版权**：正文标注“本文首发于 Upstream Labs”。



### 🔴 小红书 (视觉与情绪引导)

* **调性**：视觉冲击力 > 文字逻辑。
* **死命令**：
* **封面**：必须是代码在黑色背景下的高亮截图，或酷炫的 UI 动图。
* **排版**：善用 Emoji（如：🚀, 🔥, 🦀），段落极短。
* **标签**：必须带上 `#Rust`, `#程序员`, `#UI设计`。


### 🐦 X (Twitter) (全球技术圈)

* **调性**：快节奏、高频率、实时性。
* **死命令**：
* **Thread 模式**：长内容拆分为 3-5 条线程，第一条带视频/动图。
* **互动**：艾特相关的技术大佬（如 Makepad 作者）。

## 7. 分发后的“运营逻辑”

* **回复策略**：
* 赞美类：简要感谢。
* 质疑类：**重点回复**。用数据和 Decision Log 回击或承认不足，这最能体现专业度。


* **反馈回收**：
* 发现有价值的反馈（例如：“你们的并发逻辑在 ARM 架构下可能有坑”），必须立即同步到 `distribution-log.md` 的【反馈池】。
